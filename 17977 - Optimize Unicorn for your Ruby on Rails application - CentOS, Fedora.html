Optimize Unicorn for your Ruby on Rails application - CentOS, Fedora

Difficulty: 3

Time: 1.5 hour

Rails is a nice web framework, but over past few years, it has grown into a complex system. Particularly when using Unicorn, Rails apps tend to consume a lot of memory. Understanding its operational details and optimizing its performance can be a critical factor for success any Rails developer. This article will assist you with few simple things that can be done to enhance the performance of Unicorn for your Rails applications.

Note: It is assumed that you are logged in as a sudo user and deployed a Rails App with Unicorn and NGINX - CentOS or deployed a Rails App with Unicorn and NGINX - Fedora

Set Unicorn as a default option

By default, Rails uses the "Webrick" server to run applications. It is fast, but unfortunately, it can not handle concurrent connection requests. For optimized performance, Unicorn must be set as the default server for Rails. Unicorn is a multi-threaded web server, capable of handling multiple requests simultaneously, thus optimizing the performance 2 to 3 times.

Upgrade to 2.0

If you are using ruby 1.9 or earlier, it is recommended to upgrade to 2.0. Because ruby 1.9 and earlier versions do not support automated garbage collection, and thus are unable to take benefits of Copy-on-Write (CoW) feature.

You can check your Ruby version using "ruby -v" command. If your version is >=2.0, you are on right track, but in case it is 1.9 or earlier, you should upgrade to >=2.0

Fine tuning the configuration settings

To boost the performance of Unicorn, some configuration settings can be checked. These are available in the Unicorn's default configuration file config/unicorn.rb. Upon a new install, this configuration file might not be available on your machine. GitHub do provide a sample unicorn.rb configuration file. Checking out the following configurations can help improving the performance:

worker_process

Worker process defines the total number of concurrent worker to be launched. This ideally depends upon the memory available to the processes to execute. So depending upon your application and the availability of memory, you can set the number of concurrent worker process to run. For example, to allow 5 concurrent connections, you can set the value to 5 workers. The default value is 3.

worker_processes 3

Timeout

Timeout defines the time duration to wait before the worker times out. It is usually preferred to keep a lower base value for timeout to avoid holding back a long queue. So ideally 15 to 30 is considered as a rational value, but depending upon the nature of applications, this can be changed accordingly. The default value is 60 seconds.

timeout 60

Preload_app

This defines the ability to preload the Rails application using CoW, before forking the worker processes. This setting impacts startup time when running Unicorn worker processes. This should be set to true. Although, when using this option, special care need to be taken when opening and closing any sockets. When workers are forked, any associated connection (like database connections or others which require sockets) must be closed and reopened.

preload_app true

Controlling Unicorn Workers

When using Unicorn, all the forked or child processes of Ruby application consumes memory. So in case there is any memory leakage in the Ruby application, use of Unicorn would increase the problem multi-fold times. Over a period of time, I situation may arrive when all the memory might get used and no additional memory is left, ultimately causing the system to halt or crash. Although Unicorn may not be causing this problem, but can be resolved using Unicorn-worker-killer.

Unicorn Worker Killer

Unicorn worker killer is a gem that provides automatic restart of worker without affecting any request. It is based on the maximum number of requests and the consumed memories, and helps avoid excess consumption of memory in an event of memory leakage. Here are the steps for using unicorn-worker-killer:

  1. Installation

    gem install unicorn-worker-killer

  2. Setup Unicorn Worker Killer

    Open file config.ru (of your application) located in the default project directory, and add the following lines above the require ::File.expand_path('../config/environment', __FILE__) line.

      # --- Start of unicorn worker killer code ---

  require 'unicorn/worker_killer'

  max_request_min = 500
  max_request_max = 600

  # Max requests per worker
  use Unicorn::WorkerKiller::MaxRequests, max_request_min, max_request_max

  oom_min = (240) * (1024**2)
  oom_max = (260) * (1024**2)

  # Max memory size (RSS) per worker
  use Unicorn::WorkerKiller::Oom, oom_min, oom_max, ,versbose=true

  # --- End of unicorn worker killer code ---

  require ::File.expand_path('../config/environment',  __FILE__)
  run YourApp::Application
    max_requests_min and max_requests_max defines the minimum and the maximum number of requests per worker. To prevent all workers getting dead simultaneously, a random number between max_requests_min and max_requests_max is defined as the actual limit. Once the number exceeds the limit, that worker is automatically restarted.

Conclusion

Unicorn is painless way to achieve concurrency and performance in your Rails apps whether used with thread safe or not. It provide essential stability to Rails Apps. Unicorn worker killer gracefully restart your application and resolve memory leaks.
